# Psyche System Implementation Plan

## Goal

Add structured internal state management to phylactery — a concern graph, commitment ledger, subconscious pass, and structured briefings — giving the agent genuine continuity across sessions beyond freeform SOUL.md prose.

## Design Decisions

### What we adopt from the spec

1. **Concern graph** with typed concerns (epistemic/appetitive/conative), salience scores, and session-based decay
2. **Commitment ledger** with hard accountability — broken commitments persist in every briefing until reported
3. **Subconscious pass** — a separate binary that runs after every session, producing a structured briefing
4. **Structured briefings** — injected into the system prompt alongside SOUL.md, giving the agent explicit orientation
5. **Structured escalation** — extending `ask_human` with urgency/kind metadata rather than duplicating it
6. **Structured KB records** — subject/predicate/object triples alongside the freeform knowledge base

### What we skip

- **`pause_session` / `resume_session`** — `sleep_until` already covers the core use case. Pause would require invasive changes to phyl-run's loop and phylactd's session tracking for marginal benefit when sessions are cheap to start.
- **`watch` / `unwatch`** — Already solved by `phyl-poll` and `phyl-listen` with richer implementations (HMAC, inotify, SSE reconnection, debouncing). Instead, concerns can reference poll/listen configs by name.
- **`begin_session` / `close_session` as agent-called tools** — These become implicit in phyl-run. The runner handles session accounting automatically: briefing injection at start, obligation enforcement at end. This is more reliable than requiring model cooperation.

### Key architectural choices

- **Two new crates**: `phyl-tool-psyche` (server-mode tool) and `phyl-psyche-sub` (subconscious pass binary)
- **SQLite storage** in `$PHYLACTERY_HOME/psyche.db` — single file, no server, fits existing state model
- **Session numbers** are a monotonically incrementing counter stored in the DB, not derived from UUIDs
- **Briefing is a JSON file** at `$PHYLACTERY_HOME/briefing.json`, regenerated by the subconscious pass, read by phyl-run at session start
- **Obligation enforcement** happens in phyl-run's finalization step, not as a tool — phyl-run checks for unreported commitments and undispositioned flagged concerns before allowing clean exit
- **Escalation extends `phyl-tool-session`** rather than creating a separate tool — adds `escalate` alongside `ask_human` and `done`

## New Crates

### `phyl-tool-psyche` (server-mode tool)

A server-mode tool discovered by phyl-run via `--spec`, just like `phyl-tool-session` and `phyl-tool-self`. Provides the agent with structured internal state management.

**Tools exposed:**

| Tool | Description |
|------|-------------|
| `open_concern` | Open a new concern with type, description, optional tension, salience, tags |
| `touch_concern` | Return to a concern — record what's new, optionally adjust salience |
| `resolve_concern` | Close the gap — record outcome, optionally spawn new concerns |
| `abandon_concern` | Explicitly drop a concern with a reason |
| `surface_concerns` | Query top concerns by salience, type, state, tags |
| `commit_to` | Declare a concrete action on a conative concern with a deadline |
| `report_commitment` | Report a commitment as fulfilled or broken with a note |
| `kb_record` | Store a structured fact (subject/predicate/object/confidence/source) |
| `kb_retrieve` | Query structured facts |
| `kb_invalidate` | Mark a structured fact as invalid with a reason |

**Implementation:**

- Rust binary at `crates/phyl-tool-psyche/`
- Depends on `phyl-core` and `rusqlite`
- `--spec` returns array of `ToolSpec` with `mode: "server"`
- `--serve` reads NDJSON `ServerRequest` from stdin, writes `ServerResponse` to stdout
- Opens `$PHYLACTERY_HOME/psyche.db` on startup, creates tables if needed
- Session number comes from environment variable `PHYLACTERY_SESSION_NUMBER` (set by phyl-run)

### `phyl-psyche-sub` (subconscious pass)

A standalone binary invoked by phyl-run after SOUL.md finalization. Runs decay, generates the briefing, and exits.

**Implementation:**

- Rust binary at `crates/phyl-psyche-sub/`
- Depends on `phyl-core` and `rusqlite`
- Invoked as `phyl-psyche-sub` (no arguments — reads `$PHYLACTERY_HOME` from env)
- Steps:
  1. Open `psyche.db`
  2. Increment session counter
  3. Run salience decay on all open/committed concerns untouched this session
  4. Flag concerns below abandonment threshold
  5. Cluster open concerns by tags for suggested tensions (simple tag-overlap heuristic, not ML)
  6. Assemble briefing: top concerns, pending commitments, broken commitments, flagged concerns, suggested tensions, open escalations
  7. Write `$PHYLACTERY_HOME/briefing.json`
  8. Exit

**Decay formula:**
```
new_salience = salience * (0.5 ^ (sessions_since_touch / half_life_sessions))
```
Default half_life_sessions = 10, abandonment_threshold = 0.05. Configurable via `[psyche]` in config.toml.

### `phyl-run` — Session integration

Changes to `run_session()`:

1. **At session start** (after reading SOUL.md, before building system prompt):
   - Read `$PHYLACTERY_HOME/briefing.json` if it exists
   - Format briefing into a section of the system prompt (after SOUL, before tool list)
   - Read current session_number from briefing and set `PHYLACTERY_SESSION_NUMBER` env var

2. **In `build_system_prompt()`**:
   - Add new parameter for briefing content
   - Insert briefing section between SOUL and available tools:
   ```
   === BRIEFING ===
   {formatted briefing: top concerns, pending/broken commitments, flagged concerns, tensions}
   ```

3. **After SOUL.md finalization** (step 11, after `finalize_soul()`):
   - Invoke `phyl-psyche-sub` as a child process
   - Log success/failure to stderr
   - This runs decay and regenerates `briefing.json` for the next session

### `phyl-tool-session` — Escalation extension

Add `escalate` tool alongside existing `ask_human` and `done`:

- Takes: subject, body, urgency, kind, concern_id (optional), commitment_id (optional), blocking_action (optional), proposed_resolution (optional)
- Writes escalation to `psyche.db`
- For `blocked` and `decision_required` kinds: also calls the existing `ask_human` mechanism to notify the operator via bridge
- Returns escalation_id

This keeps operator communication unified through the existing bridge infrastructure while adding structured metadata.

## New Workspace Dependency

Add `rusqlite` to `Cargo.toml` workspace dependencies:

```toml
rusqlite = { version = "0.32", features = ["bundled"] }
```

The `bundled` feature compiles SQLite from source, avoiding system library dependencies. This is the standard approach for Rust projects.

## Storage Schema

`$PHYLACTERY_HOME/psyche.db` (SQLite):

```sql
CREATE TABLE IF NOT EXISTS concerns (
    concern_id       TEXT PRIMARY KEY,
    description      TEXT NOT NULL,
    type             TEXT NOT NULL CHECK(type IN ('epistemic','appetitive','conative')),
    tension          TEXT,
    state            TEXT NOT NULL DEFAULT 'open' CHECK(state IN ('open','committed','resolved','abandoned')),
    salience         REAL NOT NULL DEFAULT 0.5,
    tags             TEXT NOT NULL DEFAULT '[]',  -- JSON array
    origin           TEXT NOT NULL DEFAULT 'session',
    touch_count      INTEGER NOT NULL DEFAULT 0,
    created_session  INTEGER NOT NULL,
    touched_session  INTEGER NOT NULL,
    created_at       TEXT NOT NULL,
    touched_at       TEXT NOT NULL,
    resolved_at      TEXT,
    abandoned_at     TEXT,
    outcome          TEXT,
    abandon_reason   TEXT,
    spawned_from     TEXT REFERENCES concerns(concern_id),
    spawned          TEXT NOT NULL DEFAULT '[]'  -- JSON array of concern_ids
);

CREATE TABLE IF NOT EXISTS commitments (
    commitment_id    TEXT PRIMARY KEY,
    concern_id       TEXT NOT NULL REFERENCES concerns(concern_id),
    action           TEXT NOT NULL,
    scheduled_for    TEXT NOT NULL,
    fallback         TEXT,
    state            TEXT NOT NULL DEFAULT 'pending' CHECK(state IN ('pending','fulfilled','broken')),
    created_at       TEXT NOT NULL,
    reported_at      TEXT,
    note             TEXT,
    spawned_concerns TEXT NOT NULL DEFAULT '[]'  -- JSON array
);

CREATE TABLE IF NOT EXISTS escalations (
    escalation_id    TEXT PRIMARY KEY,
    subject          TEXT NOT NULL,
    body             TEXT NOT NULL,
    urgency          TEXT NOT NULL DEFAULT 'normal',
    kind             TEXT NOT NULL,
    concern_id       TEXT REFERENCES concerns(concern_id),
    commitment_id    TEXT REFERENCES commitments(commitment_id),
    blocking_action  TEXT,
    proposed_resolution TEXT,
    created_at       TEXT NOT NULL,
    responded_at     TEXT,
    response         TEXT
);

CREATE TABLE IF NOT EXISTS kb_records (
    record_id        TEXT PRIMARY KEY,
    subject          TEXT NOT NULL,
    predicate        TEXT NOT NULL,
    object           TEXT NOT NULL,
    confidence       REAL NOT NULL,
    source           TEXT NOT NULL,
    concern_id       TEXT REFERENCES concerns(concern_id),
    created_at       TEXT NOT NULL,
    expires_at       TEXT,
    invalidated_at   TEXT,
    invalidation_reason TEXT
);

CREATE TABLE IF NOT EXISTS sessions (
    session_number   INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id       TEXT NOT NULL,
    began_at         TEXT NOT NULL,
    closed_at        TEXT
);

CREATE TABLE IF NOT EXISTS briefings (
    briefing_id      TEXT PRIMARY KEY,
    session_number   INTEGER NOT NULL,
    generated_at     TEXT NOT NULL,
    content          TEXT NOT NULL  -- JSON blob of Briefing struct
);

CREATE INDEX IF NOT EXISTS idx_concerns_state ON concerns(state);
CREATE INDEX IF NOT EXISTS idx_concerns_salience ON concerns(salience DESC);
CREATE INDEX IF NOT EXISTS idx_commitments_state ON commitments(state);
CREATE INDEX IF NOT EXISTS idx_kb_records_subject ON kb_records(subject);
CREATE INDEX IF NOT EXISTS idx_escalations_responded ON escalations(responded_at);
```

## What This Enables

After implementation, the agent's session lifecycle becomes:

```
[phyl-run starts]
  → read SOUL.md + briefing.json
  → system prompt includes: identity + briefing (top concerns, pending commitments, flagged items)
  → agent works, using psyche tools to track what it cares about
  → SOUL.md reflection (existing)
  → phyl-psyche-sub runs: decay, briefing generation
  → briefing.json ready for next session
[phyl-run exits]
```

The concern graph gives the agent structured memory of what matters. Salience decay ensures neglected items surface for explicit disposition. Commitments create accountability. The subconscious pass synthesizes patterns the task-focused waking agent misses. And it all fits cleanly into phylactery's existing Unix process model — no daemon changes, no protocol changes, just two new binaries and a SQLite file.
